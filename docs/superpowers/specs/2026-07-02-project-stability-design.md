# Gift AI Enterprise 稳健治理设计

## 目标

在保持现有 API 与业务行为兼容的前提下，清理仓库环境，修复确定性缺陷，增强代码结构与安全性，并完成可验证的兼容性依赖升级。

## 当前基线

- 项目主体为 Python 3.12、FastAPI、SQLAlchemy、Redis 与 DeepSeek/OpenAI 兼容客户端。
- 工作区包含约 43 个尚未提交的 AI 模块改动，必须作为用户资产原样保护。
- 项目使用本地 `.venv`，但该目录和部分 Python 缓存已被 Git 跟踪。
- 本地数据库、上传文件与环境文件存在于工作区，不应作为普通源码进入版本控制。
- 应用能够通过项目虚拟环境启动，`/health` 返回 HTTP 200。
- 当前测试基线为 10 项中 8 项通过、2 项失败：产品分析响应字段不匹配，以及提供商路由器缺少兼容注册接口。

## 范围

### 包含

1. 仓库卫生：取消生成物、本地环境、本地数据库和上传内容的 Git 跟踪，同时保留本地实际文件。
2. 环境可复现：整理忽略规则、依赖清单、环境变量模板和统一启动方式。
3. 确定性修复：针对可复现失败建立回归测试，逐项修复根因。
4. 代码稳健性：检查异常处理、导入副作用、重复实现、未使用入口、类型边界和配置安全。
5. 兼容升级：仅升级经测试验证的补丁或次版本依赖，不主动跨主要版本。
6. 文件整理：确认无引用、无数据价值且不属于用户未提交源码的错位或无效文件可以移动或删除。
7. 源码治理：允许修改现有源码并进行有测试保护的局部重构，以改善模块边界、逻辑清晰度和运行稳定性。

### 不包含

- 改变现有 API 路径、请求模型或成功响应语义。
- 未经迁移方案修改数据库结构或删除业务数据。
- 替换 AI 提供商、缓存体系或持久化架构。
- 为追求“最新”而进行未经回归验证的主要版本升级。
- 删除有效上传文件、有效本地数据库或用户未提交业务代码。

## 治理顺序

### 第一层：仓库与环境

- 完善 `.gitignore`，覆盖虚拟环境、缓存、密钥文件、本地数据库、上传内容、测试缓存与 IDE 状态。
- 使用 Git 索引移除方式取消已跟踪生成物；不删除磁盘上的对应文件。
- 检查空入口、重复初始化脚本和错位文件；只有在引用扫描与运行验证均证明无效后才移动或删除。
- 提供不含真实密钥的 `.env.example`，并确保项目可由依赖清单重建。

### 第二层：测试与逻辑

- 固化现有测试基线，区分真实单元测试与依赖外部 Redis、数据库或网络的诊断脚本。
- 针对每个失败先确认根因，再编写或收紧失败测试，然后实施单一最小修复。
- 优先恢复已有兼容接口和响应约定，不借修复之机改变产品行为。

### 第三层：结构与安全

- 检查宽泛异常吞噬、导入时打印、重复服务入口和不清晰的模块职责。
- 对可恢复错误记录必要上下文；对不可恢复错误保持明确传播，避免静默失败。
- 检查配置默认值、密钥加载、文件上传路径、认证与数据访问边界。
- 仅做与稳定性直接相关的局部拆分，不进行全盘架构重写。

### 第四层：依赖与回归

- 对照官方发布信息识别当前兼容版本，按依赖族分批升级。
- 每批升级后执行导入检查、完整测试和应用健康检查；失败即回退该批变更并分析原因。
- 更新依赖清单，使全新虚拟环境能够确定性安装。

## 验证标准

- 项目虚拟环境可从依赖清单重建。
- 后端可启动，`GET /health` 返回 HTTP 200 和 `{"status": "healthy"}`。
- 自动化测试全部通过；外部服务不可用时，相关测试必须明确跳过或以隔离替身运行，而不是误报通过。
- Python 源码可完整编译，应用入口可导入。
- Git 状态不再因 `.venv`、`__pycache__`、本地数据库或上传文件产生噪声。
- 仓库中不包含真实密钥或环境凭据。
- 原有 API 路径和已覆盖的请求/响应行为保持兼容。

## 变更安全规则

- 每次只处理一个可解释的问题，并在继续前验证。
- 源码改动必须由可复现问题、明确结构收益或兼容升级需要驱动，并有对应测试或运行验证。
- 每个修复批次先执行针对性测试，再执行完整回归；发现新增失败时停止该批次并定位根因。
- 不使用强制重置，不覆盖用户未提交源码。
- 删除前执行引用搜索并记录理由；不确定时保留。
- 对数据库、上传内容和密钥文件只治理版本控制边界，不做数据清理。
- 任何无法保持兼容的升级都从本轮范围剔除并单独报告。
